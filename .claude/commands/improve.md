## 改善サイクル

### 1. 監査 (Audit)
Explore エージェントを使い、以下の観点でコードベースを網羅的にスキャンする:

- **SEO**: metadata / openGraph / twitter / JSON-LD の漏れ・不備、年号の古さ
- **アクセシビリティ**: ARIA属性、キーボード操作、スクリーンリーダー対応
- **内部リンク**: フッター・ヘッダー・CTA・RelatedContent の導線漏れ、回遊率改善
- **UX/UI**: モバイル表示崩れ、タップターゲット、フォーム操作性
- **パフォーマンス**: 不要な Client Component、バンドルサイズ、画像最適化
- **コンテンツ品質**: 古い情報、誤字脱字、表示崩れ
- **コード品質**: 型安全性、未使用コード、重複ロジック

### 2. 優先度判定 (Prioritize)
発見した改善項目を以下の基準で分類:

| 優先度 | 基準 | 例 |
|--------|------|----|
| P0 | バグ・表示崩れ・SEO致命傷 | テーブル崩れ、metadata欠落 |
| P1 | 回遊率・CVR直結 | CTA追加、内部リンク強化 |
| P2 | 品質向上 | アクセシビリティ、コード整理 |
| P3 | あると良い | 細かいUI改善 |

P0 → P1 → P2 の順に、1回のサイクルで対応可能な範囲を実装する。

### 3. 実装 (Implement)
- 改善をタスクリストで管理しながら実装
- 各改善は論理的にまとまったコミット単位で進める
- 既存の動作を壊さないよう注意

### 4. 検証 (Verify)
- `npm run build` でビルドが通ることを確認
- 変更内容に応じてブラウザでの確認が必要な場合はその旨を報告

### 5. デリバリー (Ship)
- 改善内容をコミット＆push
- 何を改善したかをサマリーで報告

### 6. 次サイクルへの引き継ぎ
- 今回見送った P2/P3 の項目があれば最後に列挙する
- 次回 `/improve` 実行時にそこから着手できるようにする

## 制約
- アフィリエイトリンクのURLは変更しない
- 計算ロジック（assumptions.ts）は根拠なく変えない
- 既存ページのURLは変更しない
- 1サイクルで大きすぎる変更（新ページ追加等）は提案のみ行い、実装は承認を求める
